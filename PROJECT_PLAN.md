# 向量引擎管理工作台项目规划

> 说明：本文档保留早期产品规划和阶段路线，部分“当前阶段判断”已经被 MVP 实现推进所覆盖。当前真实实现状态请以 [PROJECT_STATUS.md](PROJECT_STATUS.md) 和 [DATA_FLOW.md](DATA_FLOW.md) 为准。

## 1. 项目概述

### 1.1 项目定位
向量引擎管理工作台是一个面向基金客户服务团队的 AI 项目管理系统，核心目标是把“客户需求收集、任务拆解与分配、项目进度跟进、财务报价与结算”串成一条闭环流程，提升交付效率、降低项目延期率，并沉淀可复用的经营数据。

本项目有一个非常关键的业务特点：`项目部门按需求管理，财务部门按报价单结算`。因此系统不能只做项目协同，还必须解决`需求与报价单之间的适配、映射、核对和追踪`问题。

### 1.2 当前阶段判断
基于仓库内现有原型与流程图，项目目前处于`产品概念验证 / 方案规划`阶段，已经明确了：

- 核心用户角色：客户、项目经理、执行员工、管理者、财务
- 核心业务链路：需求收集 -> 任务指派 -> 进度跟进 -> 需求/报价适配 -> 财务结算
- 核心能力方向：AI需求解析、智能任务分配、风险预警、自动周报、智能报价、飞书深度集成
- 关键业务难点：需求口径与报价口径不一致，需要建立需求项、工作项、报价项三者的映射关系

当前尚未看到正式代码工程、数据模型、接口定义与部署方案，说明下一步重点应放在`MVP范围收敛 + 技术底座搭建 + 首批业务闭环打通`。

## 2. 业务目标

### 2.1 产品目标
- 建立统一项目工作台，替代分散在飞书文档、群消息、表格中的项目信息
- 用 AI 降低项目经理在需求整理、任务拆解、周报生成、报价核算上的重复劳动
- 提升团队任务分配合理性，减少因负载不均造成的延期
- 打通项目交付与财务结算数据，为经营分析提供基础
- 建立需求到报价单的映射规则，减少项目部与财务之间反复对表

### 2.2 可量化目标
- 需求转任务的人工处理时间下降 50%
- 项目周报生成时间下降 80%
- 高风险任务提前识别率达到 70% 以上
- 报价单生成时长从小时级下降到分钟级
- 需求与报价单适配确认时间下降 60%
- 飞书相关业务动作同步成功率达到 95% 以上

## 3. 用户与核心场景

### 3.1 用户角色
- 客户：提交需求、确认方案、查看交付与报价结果
- 项目经理：需求评估、任务拆解、资源分配、进度跟进、风险处理
- 执行员工：接收任务、更新状态、记录工时、反馈阻塞
- 管理者：查看项目全局、资源负载、经营分析
- 财务：审核报价单、核算成本、执行结算

### 3.2 核心场景
1. 客户通过飞书文档、消息或表单提交需求，系统自动抽取结构化信息
2. 项目经理确认需求后，AI 辅助拆解任务并给出工期建议
3. 系统结合员工技能、历史表现、当前负载推荐任务指派方案
4. 执行过程中自动监测延期风险，生成提醒和周报
5. 项目结束或阶段交付后，系统将需求项、任务项、工时与报价项进行匹配，生成报价单与结算数据
6. 管理层通过报表查看交付效率、资源负载、利润情况与业务建议

### 3.3 关键业务矛盾
- 项目部管理的是需求、任务、进度，不一定天然对应财务报价口径
- 财务侧只有报价单，缺少对需求变更、执行范围、交付过程的直接感知
- 同一张报价单可能覆盖多个需求项，一个需求项也可能拆成多个报价项
- 需求在执行过程中会变更，容易导致“做了但没报”或“报了但范围不清”

因此系统需要增加一个中间层：`需求-报价适配层`。

## 4. MVP 范围

### 4.1 P0：首期必须完成
- 统一登录与基础权限模型
- 项目管理：项目创建、状态流转、项目总览
- 需求管理：需求录入、需求列表、AI 结构化解析、需求拆解建议
- 任务管理：任务创建、分配、状态更新、负责人视图、工时记录
- 进度跟进：看板视图、项目风险标记、自动周报生成
- 财务结算：报价单草稿生成、人工审核、导出
- 需求报价适配：需求项与报价项映射、未匹配项提示、变更差异提示
- 飞书基础集成：文档同步、任务同步、机器人通知
- 报表基础版：项目数、任务数、逾期数、收入、负载分布

### 4.2 P1：MVP后优先增强
- 智能任务分配评分模型
- 飞书日历同步
- 飞书审批流对接报价单审核
- 多维经营报表
- 项目模板与行业模板
- 客户反馈与满意度回收
- 报价模板与行业价格库
- 变更单与补充报价流程

### 4.3 P2：中长期扩展
- 自动排期优化
- 项目利润预测
- 人力容量预测与招聘建议
- 合同管理闭环
- 多组织 / 多部门隔离
- 私有知识库驱动的企业级 AI 助理

## 5. 非功能要求

- 权限安全：按角色控制项目、任务、财务数据可见范围
- 审计追踪：需求变更、任务指派、报价修改需保留操作日志
- 可用性：核心列表页与工作台响应时间控制在 2 秒内
- 集成稳定性：飞书同步具备失败重试、告警与手动补偿能力
- AI 可控性：AI 生成内容必须支持人工确认，不直接自动写入关键业务状态
- 数据可追溯：AI 输出需保留原始输入、解析结果、确认结果

## 7. 技术方案建议

### 7.1 总体架构
- 前端：React + TypeScript + 组件库
- 后端：Node.js/NestJS 或 Java/Spring Boot
- 数据库：PostgreSQL
- 缓存 / 队列：Redis
- 文件与附件：对象存储
- AI 编排：LLM 调用层 + Prompt 模板 + 结构化输出校验
- 集成层：飞书开放平台 API + Webhook + 定时同步任务

### 7.2 AI能力落地建议
- AI需求解析：将飞书文档/消息转成结构化需求字段、优先级、功能点、建议工期
- 智能任务分配：基于技能标签、当前负载、截止时间、历史交付质量进行评分
- 风险预警：用规则引擎先落地，后续再叠加模型预测
- 周报生成：基于项目状态、工时、延期任务、阻塞项生成标准模板
- 智能报价：结合工时、角色单价、历史项目数据生成报价草稿
- 需求报价适配：AI 先将需求拆成可计价项，再与报价模板或历史报价项进行匹配，输出“已匹配 / 待确认 / 超范围新增”结果

### 7.3 需求与报价适配机制
建议在系统中单独设计一层适配逻辑，而不是直接把需求和报价单硬绑定。

适配链路建议如下：

1. 客户需求进入系统后，被拆分为`RequirementItem`
2. 项目经理确认哪些需求项进入当前交付范围
3. AI 或规则引擎将需求项匹配到标准报价模板或历史报价项
4. 财务审核匹配结果，生成`QuotationItem`
5. 系统记录`RequirementQuotationMapping`，形成可追溯关系
6. 后续若需求变更，系统自动提示哪些报价项需要补充、删除或调整金额

适配结果至少要支持以下状态：
- 已匹配
- 部分匹配
- 待人工确认
- 超范围新增
- 已废弃

### 7.4 研发策略
- 先规则、后智能
- 先人工确认、后自动闭环
- 先打通单项目流程，再扩展多项目经营分析
- 先支持“需求项 <-> 报价项”一对多/多对一映射，再逐步做自动优化

## 8. 项目里程碑

### 阶段一：方案冻结（1-2周）
- 输出 PRD、原型、角色权限矩阵、核心流程图
- 明确飞书集成边界、AI输入输出格式、MVP验收标准

### 阶段二：技术底座与基础模块（2-4周）
- 初始化前后端工程
- 完成用户、角色、项目、需求、任务、报价映射基础数据模型
- 打通基础登录、权限、项目列表、需求列表、任务列表

### 阶段三：核心业务闭环（4-8周）
- 上线需求解析、任务管理、进度跟进、周报生成
- 打通飞书文档同步、任务同步、通知
- 完成报价单草稿生成、需求映射核对与审核导出

### 阶段四：试运行与优化（8-10周）
- 选择 1-2 个真实项目试点
- 修复流程断点与同步问题
- 优化 AI 提示词、评分策略、报表口径
- 形成 V1 发布版本

## 9. 验收标准

### 9.1 MVP验收
- 可以从需求录入走到任务创建、执行跟进、需求报价适配、报价导出
- 项目经理能在一个系统内完成主要项目管理动作
- 飞书至少实现文档同步、任务同步、机器人通知三项能力
- AI 输出结果可编辑、可追溯、可确认
- 报表能展示基础经营与交付指标
- 每张报价单都可以回溯到对应需求项或变更项

### 9.2 业务验收
- 试点团队愿意持续使用，而不是回退到纯飞书/表格
- 项目经理每周固定动作中至少 3 项被系统明显提效
- 报价与周报场景形成稳定使用习惯

## 10. 风险与应对

- AI结果不稳定：关键流程采用结构化输出和人工确认
- 飞书接口复杂且变动：建立独立集成层与同步日志
- 范围过大：MVP 严格只做单条业务闭环，不一次性铺满所有能力
- 数据口径不统一：先定义需求项、任务项、报价项、工时、收入、利润的统一口径
- 团队采纳度不足：试点阶段优先解决项目经理最痛的 2-3 个场景
- 报价映射复杂：MVP 阶段先支持半自动匹配，保留财务人工确认入口

## 12. 结论
这个项目最适合走“项目管理 SaaS + 企业协同集成 + AI 助理”的路线，而不是一开始就做成全自动智能平台。尤其要把“项目部按需求管理”和“财务按报价单结算”之间的断层补上。首版要先证明三件事：

- 是否能稳定打通项目交付主链路
- 是否能在项目经理高频工作中产生明确提效
- 是否能把需求与报价单建立稳定、可追溯的映射关系

只要首个闭环跑通，后续再逐步把智能分配、经营分析和利润预测做深，项目会更容易落地，也更容易被团队持续使用。
